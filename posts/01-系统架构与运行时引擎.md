### 1.1 first_mes 的双重触发

```json
"first_mes": "<明月不孤独>\r\n<music>9y8rum.ogg</music>"
```

这条首消息本身不会呈现给用户——它被两条正则脚本拦截：

| 标签 | 匹配正则脚本 | 替换结果 |
|------|------------|---------|
| `<明月不孤独>` | 正则 #1「致谢」 | 一整页致谢 HTML 卡片（含"开始游玩"按钮） |
| `<music>9y8rum.ogg</music>` | 正则 #8「去除音乐标签」（markdownOnly） | 在 markdown 层被清除 |

用户点击致谢卡片中的"开始游玩"按钮后，JS 调用 `sendAIMessage()` 将 `first_mes` 替换为 `<sanmingyue><music>983zho.ogg</music>`，随即触发正则 #2「开场白」，渲染出完整的 **19 页角色创建向导**（天命轮转界面）。这是一个纯前端的渐进式角色创建流程：道号 → 阴阳 → 年岁 → 灵根 → 体质 → 百艺 → 修为 → 功法 → 法宝 → 灵兽 → 异火 → 势力 → 位置 → 状态 → 专精境界 → 背景 → 开局方向 → 总览确认 → 写入世界书。

这是角色卡的**数据骨架定义**，使用 Zod Schema 严格约束 AI 输出的变量结构。完整 Schema 定义了以下结构：

```

│   └── 天气: string

│   ├── 姓名 / 性别 / 年龄 / 灵根 / 体质 / 称号

│   │   ├── 灵石: { 下品, 中品, 上品, 极品 }
│   │   ├── 丹药: Record<名, { 数量, 品级, 描述 }> → transform: pickBy(数量>0)
│   │   ├── 材料: Record<名, { 数量, 品级 }> → transform: pickBy(数量>0)
│   │   └── 杂物: Record<名, { 数量, 描述 }> → transform: pickBy(数量>0)
│   ├── 功法
│   │   ├── 主修功法: { 名称, 品阶, 总层数, 当前层数, 描述 }
│   │   ├── 辅修功法: Record<名, { 品阶, 总层数, 当前层数, 描述 }>
│   │   └── 术法: Record<名, { 熟练度, 描述 }>
│   ├── 法宝

│   │   ├── 辅助法宝: Record<名, { 品阶, 描述 }>
│   │   └── 未装备法宝: Record<名, { 品阶, 描述 }>
│   ├── 灵兽: Record<名, { 品种, 境界, 描述 }>
│   └── 异火: Record<名, { 品级, 属性, 描述 }>
├── 男性好感度: Record<角色名, { 好感度(clamp -1000~1000), 关系阶段, 年龄, 重要事件 }>
├── 女性好感度: Record<角色名, { 好感度(clamp -1000~1000), 关系阶段, 年龄, 重要事件 }>
└── 剧情标记: Record<string, string>
```

**关键设计细节**：
- `prefault({})` / `prefault([])` 确保结构完整性——任何字段缺失时自动填充默认值，不会因 AI 漏写导致结构崩溃
- `z.coerce.number()` 允许 AI 输出字符串数字并自动转换
- `_.clamp(v, 0, 100)` / `_.clamp(v, -1000, 1000)` 通过 transform 硬性约束数值边界
- `_.pickBy(data, ({ 数量 }) => 数量 > 0)` 自动清理数量归零的物品条目
- 所有枚举字段均以空字符串 `''` 作为默认值，既允许初始为空，又防止 AI 编造枚举外的值

### 3.2 Prompt 节点关键内容摘要

#### main「蛇娘防标记（防429）」
使用多层 `{{random::}}` 宏生成随机 emoji + 中文字符 + 英文字母 + 数字组合，每次请求产生不同的首行内容，用于防止 API 端的请求指纹识别（429 限流）。

#### identifier 1「Catbox破限」
以对话体（明月与秋青子的工作准备对话）包裹的自检清单。核心是让 AI 在生成前识别并拒绝 6 类偏差：一概拒绝、回避冲突、引向正确答案、优待用户、泄露元信息、使用陈词滥调。

#### identifier 4「秋青子身份」
角色 `assistant`——这是一个 **assistant prefill**。将 AI 角色固定为"秋青子"——蛇娘秘书，含完整的说话方式示例、挨骂反应、工作态度等。

#### identifier 24「吃掉原生思维链」
又一个 `assistant` prefill：
```
好的，我都理解了。让我先想想该怎么写……
<think>
嗯，想好了！我马上开始创作：首先进行思考，然后输出需要的内容。
</think>
现在我会以<thinking>开始思考：
<thinking>
```
这段巧妙地"吃掉"了模型的原生 `<think>` 标签（如 DeepSeek 的 thinking 模式），使其进入自定义的 `<thinking>` 标签对，确保所有思考内容都在 EW 可剥离的范围内。

#### response_remove_regex
三个 Flow 统一使用：`<thinking>[\s\S]*?</thinking>`，即 EW 在解析 JSON 前自动剥离思考过程。

## 四、十条正则脚本

| # | 脚本名 | findRegex | 替换内容 | 作用域 | 关键标记 |
|---|--------|-----------|---------|--------|---------|
| 1 | **致谢** | `<明月不孤独>` | 完整致谢 HTML 页面 | markdownOnly | 首消息触发 |

| 7 | **只发送最新2楼的变量更新** | `/<UpdateVariable>[\s\S]*?</UpdateVariable>/gm` | `""` (空) | **promptOnly**, minDepth=4 | 深度 ≥4 时移除变量标签，只保留最近 2 楼 |
| 8 | **去除音乐标签** | `/<(Map\|WorldState\|command\|MapUpdate\|music)>[\s\S]*?<\/\1>/gi` | `""` (空) | markdownOnly | 清理显示层的功能标签 |
| 9 | **状态栏美化** | `<StatusPlaceHolderImpl/>` | 完整 Vue 状态栏应用（~2000行混淆JS+CSS） | markdownOnly | 渲染修仙风格状态面板 |

**正则系统的设计哲学**：
- 正则 #5 和 #9 同时匹配 `<StatusPlaceHolderImpl/>`，但作用域不同：#5 是 promptOnly（从提示词中移除），#9 是 markdownOnly（在显示层渲染成 Vue 状态栏）。这实现了"对 AI 不可见但对用户可见"。
- 正则 #7 的 `minDepth: 4` 意味着只对深度 ≥4 的历史消息生效——最近 2 楼的变量更新标签保留给 AI 看，更早的自动清除，节省 token。
- 正则 #3 和 #4 是一对互补正则：流式传输中（标签未闭合时）显示"正在变量中"动画，完成后（标签闭合时）显示"变量完成"可折叠面板。

## 五、动态内容控制器（~500 行 EJS 模板）

这是世界书条目 `动态内容控制器`（行 2327-2328）的核心，是一个 `@@preprocessing` 装饰器修饰的巨型 EJS 模板。它在每次对话发生前执行，决定哪些世界书条目被加载到 AI 的上下文中。

### 5.1 变量读取层

```javascript
// 从酒馆变量读取当前世界状态
var currentDomain = getvar('stat_data.世界定位.当前大域', { defaults: '中央神州' });
var currentArea = getvar('stat_data.世界定位.当前区域', { defaults: '' });
var currentScene = getvar('stat_data.世界定位.当前场景', { defaults: '' });
var presentCharacters = getvar('stat_data.在场人物', { defaults: {} });
var userCultivation = getvar('stat_data.用户信息.修为境界', { defaults: '' });
```

同时扫描最近 3 轮的用户消息和 AI 回复，用于关键词检测。

### 5.2 角色检测系统

控制器内嵌了一个**完整的角色别名映射表**（`characterAliasMap`），包含 **200+ 个角色名和别名** 的映射关系。例如：

```javascript
'殷冬雪': '殷冬雪', '冬雪': '殷冬雪',
'姬皓月': '姬皓月', '昭武女帝': '姬皓月',
'凰': '凰', '零号机': '凰', '刹那': '凰',
'花千落': '花千落', '千落': '花千落', '蝶仙': '花千落',
```

检测来源有三个：
1. `stat_data.在场人物` 变量（MVU 写入）
2. 最近 3 轮用户消息文本
3. 最近 3 轮 AI 回复文本

### 5.3 条件加载逻辑（七大模块）

#### 模块 1：动态事件指南（互斥加载）

共 14 种事件类型，一次只加载一个：

```
境界突破 → 动态事件_境界突破总规则 + 对应境界的突破细则
宗门战争 / 宗门大比 / 秘境 / 炼丹 / 炼器 / 奇遇 / 拍卖会 / 阵法 / 符箓 / 驭兽 / 狩猎 / 宗门职务
```

境界突破还有二级判断——根据 `userCultivation` 加载对应阶段的突破规则条目（突破_炼气转筑基 / 突破_筑基转金丹 / ... / 突破_合体至真仙）。

#### 模块 2：地图加载（按大域互斥）

```
中央神州 → 地图_中央神州
东荒妖域 → 地图_东荒妖域
北原魔土 → 地图_北原魔土
南疆巫地 → 地图_南疆巫地
西漠佛国 → 地图_西漠佛国
阴曹地府 → 地图_阴曹地府
四海 → 地图_四海
```

#### 模块 3：宗门/势力加载（按区域/场景条件）

**散修速览常驻加载**（不受条件限制）。其余按当前区域/场景条件判断，覆盖 **40+ 个宗门/势力**，每个宗门同时加载宗门条目 + 角色速览条目。例如：

```javascript
if (currentArea.includes('天剑峰') || currentScene.includes('天剑圣地')) {
  <%- await getwi('天剑圣地') %>
  <%- await getwi('角色速览_天剑圣地') %>
}
```

#### 模块 4：角色详细条目加载

遍历 `detectedCharacters` 数组，为每个检测到的角色加载其详细条目：

```javascript
for (const charName of detectedCharacters) {
  <%- await getwi(charName.trim()) %>
}
```

#### 模块 5：修为元指令加载

根据用户当前修为境界加载对应的元指令条目（炼气期 → 筑基期 → ... → 真仙境），共 10 个境界档位。

#### 模块 6：极阳天尊暗线系统

**与动态事件互斥**——有事件时暗线让路。暗线系统通过三重检测判断是否触发：

1. **角色检测**：18 个暗线核心角色（洛红尘、秦无伤、清奶、焚天、醉仙等）
2. **地点检测**：37 个暗线相关地点关键词
3. **遗迹关键词检测**：石碑、金光裂隙、轮回井、阳纹等

触发后按层级加载：
- 暗线总规则（始终加载）
- 主线 5 层涟漪（条件加载）
- 24+ 个地点碎片条目（按当前位置加载）
- 4 个散修碎片条目（按在场角色加载）
- 4 个遗迹条目（按关键词检测加载）

#### 模块 7：FloorZero 守卫

```javascript
const isFloorZero = (lastUserMessageId === undefined || lastUserMessageId === null)
    && (lastCharMessageId === undefined || lastCharMessageId === null || lastCharMessageId <= 0);
```

当 `isFloorZero` 为 true 时，仅加载 `元指令_AI权威准则` 一条，跳过所有动态加载——这是角色创建阶段的保护机制。

### 条目属性统计

- **constant: true**（蓝灯常驻）的条目：暗线条目、好感度参考、MVU更新、动态内容控制器
- **selective: true**（绿灯关键词触发）的条目：角色档案、宗门、地图
- **enabled: true**（在线）：好感度参考、动态内容控制器、[mvu_update] 条目
- **enabled: false**（离线但可被 EW/Controller 动态加载）：绝大多数角色和世界观条目
- **position: "before_char"**（角色定义之前）：大部分条目
- **use_regex: true**：所有条目均启用正则匹配
- **exclude_recursion: true** + **prevent_recursion: true**：防止条目间的递归引用

### A2. 暗线是如何做到"不让玩家提前激活"的？

暗线系统的防提前激活不是靠一个单一机制，而是一套**四层防护**叠加实现的：

**第一层：条目默认离线**

所有 44 条暗线世界书条目的 `enabled: false`——它们在 SillyTavern 的世界书面板上是灰灯，不会自动加载。唯一能加载它们的方式是动态内容控制器通过 `getwi()` 调用把它们"拉"进来。

**第二层：控制器的条件门控**

暗线系统被包裹在 `if (!eventGuideLoaded) { ... }` 内部。只有当前没有任何动态事件在执行时，暗线检测才会启动。这意味着——如果玩家正在炼丹、突破境界、打宗门大比，暗线条目**完全不会出现在 AI 的上下文中**，AI 物理上无法提到暗线内容。

```javascript
// 伪代码：暗线与事件互斥
if (!eventGuideLoaded) {
  // 只有没有事件时才检测暗线
  // 检测 18 个核心角色 / 37 个地点关键词 / 8 个遗迹关键词
  if (darklineTriggered) {
    // 加载暗线条目
  }
}
```

**第三层：碎片的分层条件加载**

即使暗线被触发了，也不是把 44 条全部灌进去。控制器精确到**只加载当前场景相关的碎片**：

- 暗线总规则：始终加载（告诉 AI 铁律）
- 起点涟漪：只在红尘酒家 + 洛红尘在场时加载
- 传说涟漪：暗线触发就加载（通用演绎规则）
- 遗迹涟漪：只在消息/回复中出现"石碑""金光""轮回""阳纹"等关键词时加载
- 真相涟漪：只在阴曹地府大域 或 秦无伤/清奶/焚天在场时加载
- 终点涟漪：只在红尘酒家 + 洛红尘同时满足时加载
- 30+ 地点碎片：按 `currentArea` 和 `currentScene` 精确匹配

这意味着：即使你在红尘酒家触发了暗线，你也看不到天剑圣地的碎片。你必须**物理上到达那个地方**才能听到那里的故事。

**第四层：暗线总规则的行为约束**

即使碎片条目加载了，AI 也不会直接把真相讲出来。暗线总规则定义了五条铁律：

1. AI 绝不主动揭示 {{user}} 是极阳天尊转世
2. 绝不出现"任务提示""线索发现"等游戏化语言
3. 每个角色只知道自己那一小块碎片
4. 碎片必须在角色"有理由提起"时自然出现
5. 洛红尘绝不说破、绝不催促

实际效果：玩家在红尘酒家和洛红尘喝酒聊天时，洛红尘可能会随口说一句"剑鸣城外有棵老槐树，讲的故事比我有趣"——这就是一个自然的引导，但玩家完全可以无视它。如果玩家去了那棵老槐树，才会听到老树精讲的远古传说碎片。

### A3. 好感度体系的运行时闭环

好感度不是一个文档上的参考表——它是一个**在 Zod Schema、世界书条目和 EW Flow 三层联动**的实时系统：

**存储层（Zod Schema）**：
```javascript
男性好感度: z.record(角色名, {
  好感度: z.coerce.number().transform(v => _.clamp(v, -1000, 1000)),
  关系阶段: z.string().prefault('素不相识'),
  年龄: z.string(),
  重要事件: z.string()
})
```
`_.clamp(v, -1000, 1000)` 是硬性约束——不管 AI 输出多大的数字，都会被钳制在 [-1000, 1000] 范围内。

**规则层（好感度参考条目）**：

这是一条 `constant: true` + `enabled: true` 的蓝灯常驻条目——**每一次** AI 生成时都能看到它。它定义了：
- 男性好感度上限是"生死之交"（900-1000），**绝对禁止爱情线**
- 女性好感度中"红颜知己"（700-800）本身就是美好终点，不一定要发展为爱情
- 从"红颜知己"到"情愫暗生"必须有四种特定契机之一
- 修仙世界感情发展以**年**计，不是几天就好上了

**执行层（EW Flow 1 角色更新 + Flow 2 记忆更新）**：

AI 在 Flow 1 中更新角色档案时，会参考好感度决定角色的行为倾向。在 Flow 2 中更新记忆时，会根据好感度决定角色用什么视角解读事件。好感度数值的变动本身由 AI 在 `<UpdateVariable>` 中输出，经 MVU 校验后写入 `stat_data`。

### A4. 正则脚本的"同一标签双重命运"

`<StatusPlaceHolderImpl/>` 这个标签同时被两条正则匹配——这不是 bug，是精心设计的信息不对称：

```
正则 #5 (promptOnly=true):  匹配 <StatusPlaceHolderImpl/> → 替换为 ""
正则 #9 (markdownOnly=true): 匹配 <StatusPlaceHolderImpl/> → 替换为 Vue 状态栏
```

**promptOnly** 意味着在组装发给 AI 的提示词时执行——AI 的上下文中这个标签被删掉了，AI 不知道有状态栏存在。

**markdownOnly** 意味着在渲染给用户看的 markdown 时执行——用户屏幕上显示出完整的修仙风格状态面板（修为、灵石、好感度等）。

结果：**用户看到状态栏，AI 看不到**。AI 不会在回复中提"你的修炼进度是 67%"这种破坏沉浸感的话，但用户随时能查看自己的数值。

### A5. assistant prefill 如何"骗过"模型的原生思维链

DeepSeek 等模型有内置的 `<think>` 标签。如果不处理，AI 会在 `<think>` 中输出几千字的内部推理，这些内容 EW 不会处理（EW 只匹配 `<thinking>`）。

identifier 24 的 assistant prefill 精确解决了这个问题：

```
好的，我都理解了。让我先想想该怎么写……
<think>
嗯，想好了！我马上开始创作：首先进行思考，然后输出需要的内容。
</think>
现在我会以<thinking>开始思考：
<thinking>
```

这段文本被注入到 `assistant` role 中——AI 认为自己已经说过这些话。于是：

1. `<think>` 已经打开并关闭了 → 模型的原生思维链被"吃掉"
2. `<thinking>` 被打开但未关闭 → AI 继续在 `<thinking>` 中输出思考
3. EW 的 `response_remove_regex: "<thinking>[\s\S]*?</thinking>"` 剥离整个思考过程
4. 剩下的就是纯 JSON 输出

### A6. 第零楼的特殊保护

`isFloorZero` 守卫确保角色创建阶段不会被世界书条目淹没：

```javascript
const isFloorZero = (lastUserMessageId === undefined || lastUserMessageId === null)
    && (lastCharMessageId === undefined || lastCharMessageId === null || lastCharMessageId <= 0);
```

当 `isFloorZero = true` 时，控制器只加载 `元指令_AI权威准则` 一条，跳过所有动态加载。这是因为第零楼是致谢页面 → 开场白（19 页角色创建向导），此时还没有世界状态，不需要也不应该加载宗门/角色/暗线等条目。
