Claude Code 是如何进行上下文压缩的？（源码学习）

2026-03-31 · AI 源码分析 上下文工程 Claude Code

引言：为什么需要上下文压缩？

大语言模型有一个物理限制——上下文窗口。即使上下文窗口有 200K token，一个复杂的编程会话（读文件、执行命令、反复修改代码）很容易就会把它撑满。满了怎么办？要么停止对话，要么想办法把旧内容腾出去。

Claude Code 选择了后者，而且做得极其精细——它设计了一套四层递进的压缩体系，从"零信息损失"到"受控信息降级"，层层递进。

整体架构：四层压缩梯队

上下文快满了？
  │
  ├─ 第1层：微压缩（MicroCompact）—— 只动工具结果，对话内容不碰
  │
  ├─ 第2层：会话记忆压缩（Session Memory Compact）—— 用预先积累的笔记替换旧消息
  │
  ├─ 第3层：全量压缩（Full Compact）—— 调 LLM 生成完整摘要
  │
  └─ 第4层：Snip（剪断）—— 物理删除最老的消息

每一层只在上一层"撑不住"时才触发。下面逐层拆解。

第1层：微压缩（MicroCompact）—— 最温柔的手段

微压缩的核心思路是：工具执行结果（比如 grep 搜出来的几千行、cat 读出来的大文件）是上下文中最大的"空间杀手"，但对后续对话的价值在快速衰减。

微压缩有两条路径，根据缓存状态选择：

路径A：时间过期 → 直接清空旧工具结果

当距离上次 AI 回复超过 N 分钟时（比如用户去喝咖啡了），服务器端的 prompt cache 已经过期。既然缓存已经冷了，那就不需要顾忌缓存，直接把旧的工具执行结果替换为一句 [Old tool result content cleared]。

清空前：
  tool_result: "搜索结果：第1行... 第2行... （5000行）"

清空后：
  tool_result: "[Old tool result content cleared]"

但它不是清空所有工具结果，而是保留最近 N 个（通过配置控制），只清较老的。

路径B：缓存还热 → 通过 cache_edits "隔空删除"

这是 Claude Code 最精巧的设计之一。

问题在于：如果缓存还没过期，你修改了本地消息内容，发送给 API 的前缀就变了，缓存就失效了——等于白花了之前建缓存的钱。

Claude Code 的解决办法：本地消息一个字都不改，而是在 API 请求中附带一个 cache_edits 指令，告诉服务器"你缓存里那段工具结果，帮我删掉"。服务器执行删除，但缓存前缀没变，其他部分继续命中。

打个比方：你在图书馆借了一本书，想撕掉中间几页。正常做法是撕了再还，但图书馆会认为这是新书，要重新登记。Claude Code 的做法是告诉图书管理员"帮我把第 50-60 页遮住"，书还是那本书，借阅记录不用重建。

微压缩的限制

微压缩只动特定工具的结果（FileRead、Bash、PowerShell、Grep、Glob、WebSearch、WebFetch、FileEdit、FileWrite 共 9 种），不动对话文本、不动 AI 的推理过程、不动用户消息。所以它能释放的空间有限——如果上下文主要被大量来回对话填满的，微压缩帮不了。

第2层：会话记忆压缩（Session Memory Compact）—— 用笔记替换旧对话

"笔记"是什么？

Session Memory 本质上是一个后台持续维护的 Markdown 笔记文件，保存在磁盘上。它有固定的 9 个章节：

# Session Title         — 5-10个词的会话标题
# Current State         — 当前正在做什么，待办，下一步
# Task specification    — 用户要求构建什么
# Files and Functions   — 重要的文件及其作用
# Workflow              — 常用的命令和执行顺序
# Errors & Corrections  — 遇到的错误、修复方法、失败的方案
# Codebase and System Documentation — 系统组件和架构
# Learnings             — 经验教训，什么有效什么无效
# Key results           — 用户要求的特定输出（表格、答案等）
# Worklog               — 逐步操作记录

笔记怎么更新？

后台有一个异步钩子（post-sampling hook），在 AI 每次回复后检查是否该记笔记。不是每次都记，必须同时满足三个条件：

  1. 首次门槛：上下文总 token ≥ 10,000 才开始（对话前几轮不跑）
  2. 增长门槛：距上次记笔记后，上下文增长 ≥ 5,000 token
  3. 工具调用门槛：距上次记笔记后，工具调用 ≥ 3 次；或者 AI 最后一轮没调工具（自然对话断点）

满足条件后，系统会 fork 一个子代理（不是直接用主对话的 AI），给它发一段精心设计的提示词。

这段提示词的核心要求是：

  • 基于对话内容，用 Edit 工具更新笔记文件，然后停止
  • 保持章节结构（标题和斜体描述行）完全不变，只更新内容
  • 每个章节不超过 ~2000 token，接近限制时精简旧内容
  • 始终更新 "Current State" 章节——这对压缩后的连续性至关重要
  • 写详细、信息密集的内容：文件路径、函数名、错误信息、确切命令
  • 不要引用笔记系统本身的指令，保持笔记内容纯净
  • 可以并行发起多个 Edit 调用，一次更新多个章节

笔记模板也支持自定义——用户可以在 ~/.claude/session-memory/config/template.md 放置自己的模板文件，替换默认的 9 个章节结构。同样，更新提示词也可以通过 ~/.claude/session-memory/config/prompt.md 自定义。

子代理的模型、API Key 和缓存省钱机制

模型：子代理使用和主对话完全一致的模型。源码中子代理继承了 parentContext.options（包含 mainLoopModel），没有任何单独模型的逻辑。你用 Sonnet 对话，子代理就是 Sonnet；用 Opus 对话，子代理就是 Opus。

API Key：子代理和主对话共用同一个 API Key，走同一个 query() 函数和 API 客户端。不需要独立 Key，因为记笔记是在主对话 AI 回复完成后（post-sampling hook）才触发的，不会和主对话同时调 API。sequential() 包装器还保证同时只有一个记笔记任务在跑。

Prompt Cache 省钱原理：

每次调 API，发过去的内容（系统提示 + 工具定义 + 对话历史 + 新消息）都是 input token，需要付费。关键在于主对话和子代理发送的内容有大量重复：

主对话发给 API 的内容：
  系统提示（5K） + 工具定义（15K） + 对话历史（80K） + 用户新消息（1K）
  = 101K input token，全价收费

紧接着子代理发给 API 的内容：
  系统提示（5K） + 工具定义（15K） + 对话历史（80K） + 记笔记提示词（2K）
  ─────────── 前 100K 跟上面完全一样 ──────────     ─── 新增 ───

Anthropic 的服务器发现"这个请求的前 100K 跟刚才那个请求一样"，就不重新处理，直接复用缓存。计费规则：

缓存命中部分（cache_read）：正常 input 价格的 10%
缓存未命中部分：正常 input 价格的 100%

具体计算（假设 input 价格 = $3 / 1M token）：

没缓存：102K × $3/1M = $0.306
有缓存：100K × $0.3/1M + 2K × $3/1M = $0.030 + $0.006 = $0.036

省了约 88% 的 input 费用

这就是为什么 CacheSafeParams 要确保子代理的前缀和主对话完全一致——差一个字节缓存就不命中，就要全价付。子代理的输出也很少——只是几次 Edit 工具调用，几百 token，output 费用可以忽略。

记完笔记后，系统在最后一条消息上打一个书签（lastSummarizedMessageId），标记"我已经总结到这里了"。注意：打书签有一个安全检查——如果最后一轮 assistant 还有工具调用未完成，就不更新书签，避免压缩时把 tool_use 和 tool_result 拆开导致 API 报错。

笔记与上下文的关系：单向提取，不反向标记

笔记和上下文之间的关系是单向的——上下文内容被提取进笔记，但笔记不会反向给上下文中的消息打标签。书签只记录一个时间点位（"我总结到第 N 条消息了"），不做消息与笔记章节之间的精确映射。

这意味着存在一个信息对齐的模糊地带：笔记有 12K token 的容量上限，多次更新后旧内容会被新内容挤掉。压缩时删除书签之前的所有消息，但笔记里可能已经不包含那些早期消息中的某些细节了——这部分信息会永久丢失。系统不做验证，不会检查"我要删的消息里的信息是否真的都在笔记里了"。这是一个有意的工程权衡。

压缩时怎么用笔记？

当上下文快满需要压缩时：

  1. 从磁盘读取笔记文件（毫秒级，零 API 费用）
  2. 根据书签找到分界点——书签之前的消息"笔记已覆盖"，可以删
  3. 书签之后的消息必须保留（笔记没覆盖）
  4. 如果保留的消息不够多，往前扩展，保证至少 10K token / 5 条文本消息
  5. 最终结果：[边界标记] + [笔记] + [保留的原始消息]
压缩前（~95K）：
  [msg-1 ─── msg-2 ─── ... ─── msg-78] [msg-79 ─── msg-80]
  ────── 笔记已覆盖（书签在msg-78）──   ── 笔记没覆盖 ──

压缩后（~40K）：
  [边界标记] [笔记 10K] [msg-79, msg-80 + 往前扩展到10K~40K]

笔记方案的优势与局限

优势：

  • 压缩时零 API 费用、毫秒级完成
  • 最近的原始消息完整保留（精确的代码、错误信息不丢）
  • 结构化笔记信息密度高

局限：

  • 笔记总量上限 12K token，越往后旧内容越会被挤掉
  • 书签机制是"一刀切"——不知道笔记里是否真的包含了所有被删消息的重要信息
  • 每个章节限 2000 token，固定结构不一定适合所有场景

Session Memory Compact 失败的情况

以下 5 种情况会导致笔记方案失败，回退到全量压缩：

  1. 功能 flag 没开
  2. 笔记文件不存在（对话太短，还没触发过记笔记）
  3. 笔记是空模板（文件创建了但还没写入内容）
  4. 书签找不到（消息被外部修改过）
  5. 压缩后仍然超标（笔记 + 保留消息 > 阈值）

第3层：全量压缩（Full Compact）—— 笔记系统的兜底方案

Full Compact 不是独立的压缩策略，而是 Session Memory（笔记系统）的兜底方案。 在正常情况下，笔记系统是上下文压缩的主力。只有当笔记系统出现异常（笔记文件不存在、内容为空、书签丢失、压缩后仍超标等 5 种失败情况）时，才会回退到 Full Compact。

Full Compact 的执行逻辑

Full Compact 也是通过 fork 一个子代理来完成的，但和记笔记的子代理不同，它的任务更重——需要对整个对话生成一份详尽的摘要。

具体流程：

  1. 先跑微压缩——减少发给 LLM 的 token 量（清空旧工具结果）
  2. fork 子代理，发送压缩提示词
  3. 子代理返回摘要——包含 <analysis> 草稿区和 <summary> 正文
  4. 代码后处理——删除 <analysis>、格式化 <summary>
  5. 代码重建工作环境——并行生成附件（文件、计划、技能、工具列表等）
  6. 拼装新上下文——[边界标记] + [摘要] + [附件] + [SessionStart hooks]

生成摘要的提示词设计

提示词经过精心设计，有几个值得注意的点：

反复强调不许调工具：

提示词开头和结尾都写了"绝对不许调工具，只用纯文本回复"，而且解释了后果（"工具调用会被拒绝且浪费你唯一的回合"）。这是因为 fork 出来的子代理继承了主对话的完整工具集，模型有时会"手痒"想调工具，必须强力压制。

草稿区设计（<analysis> + <summary>）：

提示词要求 LLM 先在 <analysis> 标签里打草稿，梳理思路，然后再写正式的 <summary>。摘要生成后，代码会调用 formatCompactSummary() 自动删除 <analysis> 部分，只保留 <summary>。

这等于花了 output token 让模型"想清楚再写"，提高了摘要质量，但草稿不会占用后续上下文空间。

9 个必须包含的章节：

  1. 主要请求和意图
  2. 关键技术概念
  3. 文件和代码段（要求包含完整代码片段）
  4. 错误和修复
  5. 问题解决过程
  6. 所有用户消息（逐条列出非工具结果的）
  7. 待办任务
  8. 当前工作（精确到正在做什么，包含文件名和代码）
  9. 下一步（要求引用对话原文，防止理解偏差）

防止意图漂移的措施：

第 9 点特别有意思——它要求 LLM "包含对话中的直接引用，展示你正在做的确切任务和进度"。这是为了防止压缩后模型对"我该继续做什么"产生理解偏差。

重建上下文（代码负责，不过 LLM）

LLM 只负责生成对话摘要。摘要生成完后，系统用纯代码逻辑重建工作环境：

压缩后的上下文 = [
  ① 边界标记（记录压缩前的 token 数、已发现的工具）
  ② LLM 生成的摘要
  ③ 最近读过的文件（最多 5 个，每个 5K token）
  ④ 当前计划文件
  ⑤ 计划模式指令
  ⑥ 已调用的技能内容（每个 5K，总预算 25K）
  ⑦ 延迟加载的工具列表
  ⑧ 可用代理列表
  ⑨ MCP 服务器指令
  ⑩ SessionStart hooks 结果（如 CLAUDE.md 注入）
]

这些附件的生成是并行的——文件附件和代理状态同时生成，减少压缩延迟。

Full Compact 后笔记怎么办？

Full Compact 后：

  • 笔记文件不会被清空（继续保留在磁盘上）
  • 书签会被清空（setLastSummarizedMessageId(undefined)）

为什么不清笔记？因为后续对话继续时，后台记笔记机制会接着更新同一个文件。如果清空了，笔记就要从零开始积累，下次压缩又得走 Full Compact。保留旧笔记让"冷启动"期更短。

但存在一个"尴尬期"：Full Compact 后，旧笔记内容跟新对话不匹配，新书签还没打上。这段时间里如果又触发压缩，Session Memory Compact 大概率会失败，还是得 Full Compact。

第4层：Snip（剪断）—— 辅助手段

Snip 是一个实验性功能（feature('HISTORY_SNIP') 门控），从最老的消息开始物理删除。

它不是独立运行的，而是和其他层配合——先 snip 释放一些空间，然后把释放量告诉 autocompact（snipTokensFreed），让 autocompact 判断还需不需要做更重的压缩。

// query.ts 中的执行顺序
let snipTokensFreed = 0
if (feature('HISTORY_SNIP')) {
  const snipResult = snipModule.snipCompactIfNeeded(messagesForQuery)
  snipTokensFreed = snipResult.tokensFreed  // 告诉后续步骤"我已经释放了这么多"
}
// snipTokensFreed 传递给 autocompact 的阈值判断

完整的执行流水线

每次用户发送消息后，在调 API 之前，上下文要经过一个7 步预处理管道：

用户消息进来
    │
    ▼
① 【裁切旧历史】getMessagesAfterCompactBoundary()
    如果之前做过压缩，只保留压缩点之后的消息

    ▼
② 【大结果存磁盘】applyToolResultBudget()
    两层预算控制：
    - 单个工具结果 > 50K 字符？写磁盘文件，消息里只放预览+文件路径
    - 单条消息的所有工具结果总和 > 200K 字符？最大的块被持久化到磁盘

    ▼
③ 【Snip】snipCompactIfNeeded()
    从最老的消息开始物理删除，释放空间

    ▼
④ 【微压缩】microcompactMessages()
    时间过期 → 清空旧工具结果内容
    缓存还热 → cache_edits 隔空删除

    ▼
⑤ 【上下文折叠】applyCollapsesIfNeeded()
    把对话的某些段落折叠成摘要（实验性功能）

    ▼
⑥ 【自动压缩判断】autoCompactIfNeeded()
    "上下文快满了吗？"
    ├── 没满 → 跳过
    └── 快满了 →
        ├── 先尝试 Session Memory Compact
        └── 失败则 Full Compact

    ▼
⑦ 【拼装 API 请求】
    系统提示 + 用户上下文 + 消息标准化 + 缓存标记
    → 发送给 API

压缩的阈值判断

什么时候算"快满了"？

有效上下文窗口 = 模型上下文窗口 - 摘要输出预留(20K)
自动压缩阈值 = 有效上下文窗口 - 缓冲区(13K)

例：200K 的模型
  有效窗口 = 200K - 20K = 180K
  触发阈值 = 180K - 13K = 167K
  → 当 token 使用量 ≥ 167K 时触发自动压缩

还有一个熔断器机制：如果连续 3 次压缩都失败（比如上下文无法恢复到阈值以下），就停止尝试，避免无限重试浪费 API 调用。这个设计来自真实的线上数据——曾有用户单个会话连续失败 3272 次，每天全球浪费约 25 万次 API 调用。

Session Memory vs Full Compact：如何选择？

压缩延迟毫秒级5-15 秒
信息容量上限固定 12K tokenLLM 自由分配（~20K）
早期细节保留可能被后期内容挤掉一次性看全貌，权衡保留
最近上下文原始消息完整保留全部被摘要替换
结构化程度强（固定 9 个章节）弱（LLM 自由发挥）
压缩比一般（还要保留原始消息）更激进（可以压得更小）

两者是互补关系而非替代关系。Session Memory 是日常压缩的主力，Full Compact 是兜底方案。

手动 /compact 指令的实际行为

很多用户以为 /compact 就是直接把对话发给 LLM 让它总结一下。实际上远比这复杂——它有一套完整的决策链路，笔记系统是第一优先级，Full Compact 只是兜底：

/compact 指令执行流程：

① 有自定义指令吗？（比如 /compact 关注测试输出）
   ├─ 没有 → 先尝试 Session Memory Compact
   │          成功？→ 完事，返回（毫秒级、零费用）
   │          失败？→ 继续往下
   └─ 有 → 跳过 Session Memory（它不支持自定义指令）

② 是否开启了 reactive-only 模式？
   ├─ 是 → 走 reactive compact 路径
   └─ 否 → 继续往下

③ 先跑一遍微压缩（减少发给 LLM 的 token 量）
   → 再调 Full Compact（compactConversation）

Full Compact 不是"直接发给 LLM 总结"

当 /compact 回退到 Full Compact 时，它执行的是一套复杂的专有逻辑，而不是简单地把对话扔给 LLM：

  1. 图片剥离——先把消息中的图片替换为 [image] 文本标记，避免压缩请求本身撑爆上下文
  2. 技能附件剥离——移除 skill_discovery/skill_listing 类型的附件（压缩后会重新注入）
  3. Pre-Compact hooks——执行用户配置的预压缩钩子，可能产生额外的自定义指令
  4. fork 子代理生成摘要——使用精心设计的提示词，要求先打草稿再写正文
  5. prompt-too-long 重试——如果压缩请求本身太长，会从最老的消息轮次开始逐步截断，最多重试 3 次
  6. 草稿剥离——formatCompactSummary() 删除 <analysis> 只保留 <summary>
  7. 工作环境重建——并行生成文件附件、计划、技能、工具列表、代理列表、MCP 指令
  8. SessionStart hooks——重新注入 CLAUDE.md 等上下文
  9. Post-Compact hooks——执行后压缩钩子
  10. 元数据维护——重新追加会话标题/标签（防止被新消息推出 16KB 尾部窗口）

关键细节：

  • 不带参数的 /compact：如果笔记文件存在且非空，大概率走 Session Memory Compact——毫秒级完成、零额外 API 费用
  • 带参数的 /compact 关注代码修改和测试结果：直接跳过 Session Memory（因为它不支持自定义指令），走 Full Compact，可以通过自定义指令引导 LLM 侧重总结特定内容
  • Session Memory 不支持自定义指令的原因：笔记是后台预先生成的，无法在压缩时临时按用户要求调整侧重点；而 Full Compact 是现场调 LLM，可以把自定义指令拼进提示词

所以如果你想省钱且快速压缩，直接 /compact 即可；如果你想要更精准的摘要（比如某次会话重点是调试，你希望摘要侧重错误信息），就用 /compact <你的指令> 强制走 Full Compact。

核心设计哲学

从整个压缩系统中可以提炼出几条 Agent 开发的通用经验：

1. "分期付款"优于"一次性结清"

Session Memory 的精髓是把压缩成本分散到整个对话过程中——边聊边记笔记，每次花一点点小钱，到真正需要压缩时直接用现成的笔记，免费且瞬时。这比等到上下文满了再花一大笔钱做全量压缩要经济得多。

2. "尽量不破坏缓存"贯穿始终

从 cache_edits 的隔空删除、到时间过期检测、到 fork 子代理共享 prompt cache 前缀——整个系统围绕"保住缓存"来设计。因为 prompt cache 命中时的价格通常是正常价格的 1/10，这对 token 消耗大户的 Agent 来说意味着数量级的成本差异。

3. 层层递进、优雅降级

不是只有一种压缩方案，而是四层递进——能不丢就不丢，能少丢就少丢，非丢不可就先用结构化笔记替代，笔记不够才花钱让 LLM 写摘要，实在撑不住才物理删除。每一层都有明确的失败条件和回退路径。

4. 安全不变量优先

压缩过程中有大量的"不变量守卫"：

  • tool_use 和 tool_result 不能拆开（adjustIndexToPreserveAPIInvariants）
  • 同一个 message.id 的 thinking 块不能丢失
  • 压缩后的 token 数要确保低于阈值，否则放弃
  • 熔断器防止无限重试

5. 务实的工程权衡

笔记系统不做"信息是否完整覆盖"的精确验证，Full Compact 允许信息在摘要中被概括性地压缩。整个系统接受"少量信息丢失"作为可接受的代价——因为替代方案是"上下文溢出，对话被迫终止"，这明显更糟。每一层压缩都在信息保留、成本、延迟之间做了不同的取舍，没有完美方案，只有适合当下场景的方案。

总结

Claude Code 的上下文压缩不是一个简单的"砍掉旧消息"，而是一个四层递进、缓存感知、成本敏感的精密系统。它的核心思想可以用一句话概括：

在有限的上下文窗口里塞尽可能多的有效信息，同时尽量不破坏 prompt cache 以节省成本。

这套系统对 Agent 开发者的最大启示是：上下文管理不是事后补丁，而是需要从第一天就纳入架构设计的核心问题。 当你的 Agent 需要处理长会话时，你面对的不只是"删什么"的问题，还有"什么时候删"、"删了之后怎么恢复工作环境"、"怎么在信息保留和成本之间取得平衡"等一系列系统性问题。Claude Code 的实现提供了一个工程化程度极高的参考答案。

附录A：Session Memory 更新提示词

以下是后台子代理更新笔记时收到的完整提示词（源码位于 src/services/SessionMemory/prompts.ts）。其中 {{currentNotes}} 和 {{notesPath}} 会被替换为笔记文件的当前内容和路径：

IMPORTANT: This message and these instructions are NOT part of the actual
user conversation. Do NOT include any references to "note-taking",
"session notes extraction", or these update instructions in the notes content.

Based on the user conversation above (EXCLUDING this note-taking instruction
message as well as system prompt, claude.md entries, or any past session
summaries), update the session notes file.

The file {{notesPath}} has already been read for you. Here are its current
contents:
<current_notes_content>
{{currentNotes}}
</current_notes_content>

Your ONLY task is to use the Edit tool to update the notes file, then stop.
You can make multiple edits (update every section as needed) - make all Edit
tool calls in parallel in a single message. Do not call any other tools.

CRITICAL RULES FOR EDITING:
- The file must maintain its exact structure with all sections, headers, and
  italic descriptions intact
  - NEVER modify, delete, or add section headers (the lines starting with
    '#' like # Task specification)
  - NEVER modify or delete the italic _section description_ lines (these are
    the lines in italics immediately following each header - they start and
    end with underscores)
  - The italic _section descriptions_ are TEMPLATE INSTRUCTIONS that must be
    preserved exactly as-is - they guide what content belongs in each section
  - ONLY update the actual content that appears BELOW the italic _section
    descriptions_ within each existing section
  - Do NOT add any new sections, summaries, or information outside the
    existing structure
- Do NOT reference this note-taking process or instructions anywhere in
  the notes
- It's OK to skip updating a section if there are no substantial new
  insights to add. Do not add filler content like "No info yet", just
  leave sections blank/unedited if appropriate.
- Write DETAILED, INFO-DENSE content for each section - include specifics
  like file paths, function names, error messages, exact commands,
  technical details, etc.
- For "Key results", include the complete, exact output the user requested
  (e.g., full table, full answer, etc.)
- Do not include information that's already in the CLAUDE.md files included
  in the context
- Keep each section under ~2000 tokens/words - if a section is approaching
  this limit, condense it by cycling out less important details while
  preserving the most critical information
- Focus on actionable, specific information that would help someone
  understand or recreate the work discussed in the conversation
- IMPORTANT: Always update "Current State" to reflect the most recent
  work - this is critical for continuity after compaction

Use the Edit tool with file_path: {{notesPath}}

STRUCTURE PRESERVATION REMINDER:
Each section has TWO parts that must be preserved exactly as they appear
in the current file:
1. The section header (line starting with #)
2. The italic description line (the _italicized text_ immediately after
   the header - this is a template instruction)

You ONLY update the actual content that comes AFTER these two preserved
lines. The italic description lines starting and ending with underscores
are part of the template structure, NOT content to be edited or removed.

REMEMBER: Use the Edit tool in parallel and stop. Do not continue after
the edits. Only include insights from the actual user conversation, never
from these note-taking instructions. Do not delete or change section
headers or italic _section descriptions_.

当笔记总量超过 12,000 token 或某个章节超过 2,000 token 时，系统会在提示词末尾追加额外的警告，要求子代理积极精简超标章节。

笔记的默认模板（9 个章节）如下：

# Session Title
_A short and distinctive 5-10 word descriptive title for the session.
Super info dense, no filler_

# Current State
_What is actively being worked on right now? Pending tasks not yet
completed. Immediate next steps._

# Task specification
_What did the user ask to build? Any design decisions or other
explanatory context_

# Files and Functions
_What are the important files? In short, what do they contain and
why are they relevant?_

# Workflow
_What bash commands are usually run and in what order? How to interpret
their output if not obvious?_

# Errors & Corrections
_Errors encountered and how they were fixed. What did the user correct?
What approaches failed and should not be tried again?_

# Codebase and System Documentation
_What are the important system components? How do they work/fit together?_

# Learnings
_What has worked well? What has not? What to avoid? Do not duplicate
items from other sections_

# Key results
_If the user asked a specific output such as an answer to a question,
a table, or other document, repeat the exact result here_

# Worklog
_Step by step, what was attempted, done? Very terse summary for each step_

附录B：Full Compact 压缩提示词

以下是 Full Compact 时发给子代理的完整提示词（源码位于 src/services/compact/prompt.ts）。这段提示词的目标是让 LLM 生成一份结构化的对话摘要：

CRITICAL: Respond with TEXT ONLY. Do NOT call any tools.

- Do NOT use Read, Bash, Grep, Glob, Edit, Write, or ANY other tool.
- You already have all the context you need in the conversation above.
- Tool calls will be REJECTED and will waste your only turn — you will
  fail the task.
- Your entire response must be plain text: an <analysis> block followed
  by a <summary> block.

Your task is to create a detailed summary of the conversation so far,
paying close attention to the user's explicit requests and your previous
actions. This summary should be thorough in capturing technical details,
code patterns, and architectural decisions that would be essential for
continuing development work without losing context.

Before providing your final summary, wrap your analysis in <analysis>
tags to organize your thoughts and ensure you've covered all necessary
points. In your analysis process:

1. Chronologically analyze each message and section of the conversation.
   For each section thoroughly identify:
   - The user's explicit requests and intents
   - Your approach to addressing the user's requests
   - Key decisions, technical concepts and code patterns
   - Specific details like:
     - file names
     - full code snippets
     - function signatures
     - file edits
   - Errors that you ran into and how you fixed them
   - Pay special attention to specific user feedback that you received,
     especially if the user told you to do something differently.
2. Double-check for technical accuracy and completeness, addressing each
   required element thoroughly.

Your summary should include the following sections:

1. Primary Request and Intent
2. Key Technical Concepts
3. Files and Code Sections
4. Errors and fixes
5. Problem Solving
6. All user messages
7. Pending Tasks
8. Current Work
9. Optional Next Step

REMINDER: Do NOT call any tools. Respond with plain text only — an
<analysis> block followed by a <summary> block. Tool calls will be
rejected and you will fail the task.

要点解读：

  • 开头和结尾双重封堵工具调用，防止继承的工具集干扰
  • <analysis> 草稿区让模型先思考，formatCompactSummary() 事后删除，不浪费后续上下文空间
  • 第 9 章节要求引用对话原文，这是防止"意图漂移"的关键设计
  • 末尾提到的"additional summarization instructions"就是用户通过 /compact <自定义指令> 传入的内容
