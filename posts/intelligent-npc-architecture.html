构建沉浸世界的智能 NPC（一）Context is all you need !

2026-03-27 · AI 游戏开发 NPC对话 架构设计

为Unity游戏构建了一套完整的AI NPC对话系统，让NPC能够根据世界观、角色人设、游戏状态进行智能对话。本文将以一个具体案例为线索，展示系统从"传统预设对话"到"完整AI对话系统"的演进过程。

核心成果

指标数值
核心子系统6个（人设卡、世界书、上下文编排、记忆、时间、调试）
Context Provider6种（Persona、AuthorNote、WorldBook、ChatSummary、VectorMemory、GameState）
Token成本优化减少50-70%（通过智能编排只注入相关片段）
查询性能优化250x-1000x（倒排索引 + Regex缓存）

案例导入：玩家与幽灵的对话

全文将以这个案例为线索，展示系统从"传统预设对话"到"完整AI对话系统"的演进过程。

场景设定：

  • 地点：废弃的古堡地下室
  • NPC：艾琳娜（300年前被囚禁于此的贵族幽灵）
  • 背景：艾琳娜生前是王国公主，因被诬陷通敌而被囚禁，死后化为幽灵，徘徊在古堡中寻找证明清白的证据

第一部分：体验演进

阶段一：传统预设对话（痛点分析）

传统的对话树实现方式：

// 传统的对话树实现
public class TraditionalDialogue : MonoBehaviour
{
    private Dictionary<string, string[]> dialogueOptions = new()
    {
        {"greeting", new[] {"你是谁？", "这里是什么地方？", "我想离开"}},
        {"response_who", new[] {"我是艾琳娜...曾经的公主。"}},
        {"response_where", new[] {"这是我被囚禁的地方..."}},
        {"response_leave", new[] {"你可以走，但我走不了..."}}
    };
}

玩家：你知道关于黑龙战争的事吗？

传统方案问题
「抱歉，我不明白你在说什么」NPC只能回应预设问题
或者干脆没有回应选项对话树无法处理开放问题

传统对话树的三大痛点：

  • 无法处理开放问题 — 玩家只能从预设选项中选
  • 缺乏记忆和连贯性 — 每次对话都是全新开始
  • 内容创作成本高 — 每个NPC需要大量对话分支

阶段二：引入AI大模型（基础能力）

玩家：你知道关于黑龙战争的事吗？

AI方案效果
「黑龙战争？那是一场可怕的战争，发生在很久以前...」能够回应开放问题了！

但新问题出现了 — AI太自由，会胡编乱造：

AI回答问题
「黑龙战争死了一百万人，国王亲自出征...」AI编造的！游戏里不是这样设定的

关键洞察：AI太自由了，我们需要约束AI只说应该说的内容——这就是世界书要做的事。

阶段三：世界书系统（约束AI的知识边界）

核心问题：如何让AI只说我们设定好的内容，不要胡编？

解决方案：创建一个"知识库"，当玩家提到某个话题时，把相关知识注入给AI。

世界书条目示例：

{
  "entryId": "wb_dragon_war",
  "title": "黑龙战争",
  "keywords": ["黑龙战争", "黑龙", "尼德霍格"],
  "content": "黑龙战争发生在280年前，持续三年。黑龙'尼德霍格'袭击王国北境。
              最终被银月骑士团击退，但王国北部十城化为废墟。
              艾琳娜的兄长阿尔贝特王子在此战中阵亡。"
}

触发机制：玩家说"你知道黑龙战争吗？" → 关键词匹配 → 注入对应知识到Prompt

效果：

有世界书效果
「黑龙战争发生在280年前，我的兄长阿尔贝特就是在那场战争中阵亡的......那头叫尼德霍格的黑龙，我至今还记得。」知识准确，与游戏设定一致！

但表达方式还是很机械 — 像在念百科全书，没有情感。

关键洞察：世界书解决了"说什么"的问题，但没解决"怎么说"的问题。我们需要给NPC注入性格和灵魂。

阶段四：角色卡系统（激活NPC性格）

核心问题：如何让NPC有独特的性格、语气、情感表达？

角色卡示例：

你是艾琳娜，300年前因被诬陷通敌而被囚禁于此的公主幽灵。

## 性格特点
- 高贵但不傲慢，对善意的人友好
- 带着深深的悲伤和执念
- 对"背叛"这个词极其敏感
- 渴望被理解和帮助

## 说话风格
- 使用古典优雅的措辞
- 常用"......"表示沉默或回忆
- 提到过去时会变得忧伤
- 偶尔流露出贵族的骄傲

效果：

玩家提问有角色卡的回答
你是怎么死的？「......我被关在这里三年，最后病死在这冰冷的牢房里。那一天，是我22岁生日......时间太久了，我有时会忘记......」
你恨陷害你的人吗？「恨？......也许曾经恨过。但300年的时间，足够让恨意消散。现在我只想......证明我的清白，然后安息。」

世界书 vs 角色卡的分工：

模块解决的问题内容类型共享性
世界书说什么（知识）事件、地点、物品、历史所有NPC共享
角色卡怎么说（性格）语气、措辞、情感、态度每个NPC独有

为什么要分开？黑龙战争是世界的客观事实，艾琳娜和酒馆老板都可能知道。但艾琳娜提到战争时会悲伤（因为兄长阵亡），酒馆老板则可能只是感慨。分开后可以复用：新增NPC只需写角色卡，关联现有世界书即可。

阶段五：私有记忆与认知限制

核心问题：如何让不同NPC知道不同的事情？NPC不应该是全知全能的。

通过角色卡中的 worldBooks 字段指定NPC能够访问的世界书条目：

{
  "npcId": 1001,
  "npcName": "艾琳娜",
  "worldBooks": [
    "wb_dragon_war",         // 公共知识：黑龙战争
    "wb_ghost_castle",       // 公共知识：古堡历史
    "wb_elina_secret_letter" // 私有知识：只有艾琳娜知道信件位置
  ]
}

效果对比：

角色问"信件藏在哪"
艾琳娜（有私有知识）「......信件就藏在东塔的密室里。那是我临死前托侍女莉莉丝藏起来的......」
酒馆老板（无此知识）「信件？什么信件？你说的是那个古堡里的幽灵公主吗？我只听说过那个传说......」

阶段六：记忆系统（维持对话连贯性）

核心问题：如何让NPC记住对话内容，即使对话历史被裁剪？

三级记忆架构：

  • L1：短期记忆 — 最近10轮对话，直接作为Messages发给AI
  • L2：对话摘要 — 压缩早期对话为摘要，保留关键信息（5-10x压缩）
  • L3：长期记忆 — 重要事实存储为JSON，跨会话保留

效果：

场景NPC回答
第20轮："你还记得我叫什么吗？"「艾德里安......当然记得。你是唯一一个说要帮我的人。」
跨会话（第二天登录）："我回来了。"「艾德里安！你真的回来了......我还以为你会像其他人一样，离开后就忘记我。」

阶段七：作者注释（运行时引导）

核心问题：如何在运行时动态引导NPC说符合当前情况的话？

作者注释（Author's Note）是一种运行时动态注入的引导信息，告诉AI"当前应该注意什么"。

{
  "authorNoteEnabled": true,
  "authorNoteContent": "当前情境：玩家刚获得东塔钥匙。你应该：
    1.表现出惊喜和希望 2.主动提到信件的事 3.引导玩家前往东塔",
  "authorNoteDepth": 2
}
无作者注释有作者注释
「你想聊什么？」「艾德里安！你手里拿的是......东塔的钥匙？！300年了，终于有人能打开那扇门了......」

常见应用场景：

场景作者注释内容
玩家首次见面"这是第一次见面，应该先自我介绍"
任务完成后"表现出感激，给予奖励，提示下一步任务"
玩家长时间未行动"委婉提醒玩家当前目标"
剧情高潮点"情绪激动，语气强烈"

阶段八：动态权重机制（解决答非所问）

核心问题：如何确保NPC的回答始终聚焦在玩家关心的话题上？

当系统注入了大量上下文，AI可能会被不相关的信息干扰。解决方案是匹配权重加成：

最终权重 = 基础优先级 × 匹配权重 × 新鲜度权重

匹配权重：
- 关键词精确匹配当前问题：1.5x
- 部分匹配：1.0x
- 无匹配：0.8x

效果（玩家问"东塔钥匙在哪"）：
├── 东塔钥匙：50 × 1.5 = 75 ← 现在最高了！
├── 黑龙战争：70 × 0.8 = 56
└── 古堡历史：60 × 0.8 = 48

防复读机制 — 新鲜度权重衰减：

  • 刚使用（1轮内）：0.2x — 大幅降低
  • 最近使用（3轮内）：0.5x
  • 较近使用（10轮内）：0.8x
  • 久未使用：1.0x

至此，体验演进完成：

  • 能回应开放问题（AI大模型）
  • 知识准确不胡编（世界书）
  • 有独特性格表达（角色卡）
  • 认知有边界（私有记忆）
  • 能记住对话内容（记忆系统）
  • 能引导玩法（作者注释）
  • 回答聚焦不跑题（动态权重）

但这一切的背后，本质上是在做一件事——上下文工程。

第二部分：工程实现

本质：LLM的上下文工程

前面所有的"魔法"，本质上都是在做一件事：在有限的Token预算内，给AI组装最合适的上下文。

LLM只是一个"补全器"，它只能基于你给它的上下文来回答。给它什么，它就用什么来回答：

输入（上下文）输出（回答）
"你是艾琳娜，一个幽灵"冷冰冰的幽灵回答
"你是艾琳娜，300年前被囚禁的公主幽灵，性格忧伤高贵......"有情感、有故事的回答
上面 + "黑龙战争的历史知识"知识准确的回答
上面 + "玩家刚完成任务，表现惊喜"推进剧情的回答

所以，AI对话系统的核心工作 = 为每次对话组装最合适的上下文。

上下文组装流程：

  1. 加载Persona — PersonaManager获取NPC人设
  2. WorldBook匹配 — 倒排索引O(1)查找相关知识
  3. 记忆检索 — 查找对话摘要 + 长期记忆
  4. 作者注释 — 检查是否有运行时引导
  5. 上下文编排 — 收集所有Provider片段，按权重排序，Token预算内裁剪
  6. 构建Prompt — 组装SystemPrompt + Messages
  7. API请求 — 流式请求，打字机效果展示

总体思路总结：从现实世界抽象而来

整个系统是对现实世界中"人"如何对话的抽象：

现实世界系统设计
世界客观存在的事实世界书（WorldBook）— 所有NPC共享的知识库
每个人独特的性格角色卡（Persona）— 决定NPC"怎么说话"
每个人知道的不一样认知限制（WorldBooks关联）— NPC只能访问关联的知识
每个人有自己的记忆记忆系统（Memory）— 摘要 + 长期记忆持久化
对话时会根据情境调整作者注释（AuthorNote）— 运行时动态引导
注意力会集中在关键话题动态权重（匹配加成）— 相关信息优先级提升

游戏世界的三层架构：

  • A. 客观事实（世界书）：独立于任何角色存在的事实。所有NPC共享，是世界的"真相"。
  • B. 人格性格（角色卡）：决定NPC"怎么说话"。每个NPC独有，是角色的"灵魂"。
  • C. 认知记忆（关联机制 + 记忆系统）：NPC不是全知全能的。限制NPC知道什么、记得什么。

完整对话流程：实际Prompt揭示

当玩家问"你知道黑龙战争吗？"时，系统最终组装出的Prompt：

# SystemPrompt（实际发送给AI的内容）

## 角色设定
你是艾琳娜，300年前因被诬陷通敌而被囚禁于此的公主幽灵。
性格特点：高贵但不傲慢，带着深深的悲伤和执念......
[来自PersonaProvider，~500 tokens]

## 作者注释
当前场景氛围：神秘、悲伤。引导玩家探索古堡秘密。
[来自AuthorNoteProvider，~50 tokens]

## 世界知识
[黑龙战争]
黑龙战争发生在280年前，持续三年......艾琳娜的兄长阵亡。
[来自WorldBookProvider，~200 tokens]

[王子阿尔贝特]
艾尔文王国王子，艾琳娜的兄长。在黑龙战争中英勇战死......
[来自WorldBookProvider递归触发，~150 tokens]

## 对话摘要
玩家自称艾德里安，是冒险者，表示愿意帮助艾琳娜证明清白。
[来自ChatSummaryProvider，~100 tokens]

## 当前状态
- 游戏时间：夜晚
- 地点：古堡地下室
[来自GameStateProvider，~30 tokens]

---
总计：~1030 tokens（预算1500内）

内容创作工作流

这套工作流的目标是让策划专注于创意工作：

传统方式：
策划 → 写Excel表格 → 手动配置关键词 → 手动导入Unity → 反复调试

新方式：
策划 → 写剧本（创意工作）→ LLM自动拆分 → 一键导入 → 监控面板调试
              ↑
      策划只需要专注这里

工作流步骤：

  1. 策划编写剧本（.md文件）— 场景描述、角色设定、对话示例、关键事件
  2. LLM辅助拆分（Coze工作流）— 自动生成角色卡JSON和世界书条目JSON
  3. Unity导入工具（Editor Window）— 选择JSON文件，预览后一键导入
  4. 监控面板调试 — 实时查看Prompt构建详情、Token消耗、Provider状态

性能优化

策略实现方式效果
上下文编排只注入相关片段Token减少50-70%
倒排索引Dictionary快速查询O(n)→O(1)
Regex缓存预编译正则表达式1000x提升
Token熔断maxTokens限制防止超限
对话摘要压缩历史对话5-10x压缩

Token成本估算：

场景优化前优化后节省
单次对话（50条WorldBook）~12,500 tokens~1,500 tokens88%
20轮对话历史~4,000 tokens~1,200 tokens70%
每日1000次对话~12.5M tokens~1.5M tokens88%

响应时长：用户感知的等待时间主要是API请求时间（~1秒），本地处理（WorldBook查询~0.2ms + 上下文编排~8ms）几乎无感知。

总结

从传统对话树到完整AI对话系统的演进路线：

  1. 传统对话树 → 无法处理开放问题
  2. 接入AI大模型 → AI会胡编乱造
  3. 世界书系统 → 约束AI的知识边界
  4. 角色卡系统 → 激活NPC性格
  5. 私有记忆 → 认知限制
  6. 记忆系统 → 维持对话连贯性
  7. 作者注释 → 运行时引导
  8. 动态权重 → 解决答非所问

最终效果：玩家可以与NPC进行自然、连贯、有深度的AI对话。NPC记得玩家，了解世界观，保持角色一致，能引导玩法。Token成本可控，开发效率高，策划专注创作。

而这一切的本质，就是上下文工程——在有限的Token预算内，为每次对话组装最合适的上下文。
