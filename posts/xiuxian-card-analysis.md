# 「修仙世界」角色卡深度拆解

> 基于 SillyTavern chara_card_v3 规范，对社区高质量修仙世界角色卡的逆向工程分析。

---

## 一、整体架构总览

这张卡不是传统意义上的"角色卡"，而是一个**完整的游戏世界运行时系统**。它的核心架构可以概括为：

```
角色卡
├── 前端脚本层（Tavern Helper Scripts）
│   ├── MVUzod —— 变量状态管理引擎
│   ├── ZOD结构脚本 —— 数据模型定义（Schema）
│   ├── 音乐播放器 —— 沉浸式BGM系统
│   └── Evolution World Assistant —— 动态世界运行时
├── 提示词模板层（Prompt Template）
│   ├── 防标记系统 —— 随机token防API限流
│   ├── 破限系统 —— 角色扮演框架
│   ├── 创作原则 —— 输出质量控制
│   ├── EW输出格式 —— 结构化JSON输出规范
│   ├── 角色动态更新系统 —— 自动更新角色档案
│   ├── 角色记忆系统 —— 主观记忆管理
│   ├── 主时间线系统 —— 客观事件记录
│   └── 多层思维链 —— 分步推理框架
├── 世界书层（World Book / Lorebook）
│   └── 443条条目 —— 角色人设、势力设定、地理、功法等
└── 数据模型层（ZOD Schema）
    ├── 世界定位 —— 场景/事件/天气
    ├── 用户信息 —— 修为/背包/功法/法宝
    ├── 好感度系统 —— 男性/女性分轨
    └── 剧情标记 —— 事件flag
```

## 二、脚本层拆解

### 2.1 MVUzod —— 变量状态管理

外部依赖：`MagVarUpdate` 库。作用是在 SillyTavern 中维护一套**结构化变量系统**，每条消息都能读写变量，实现跨轮次的状态持久化。

提供的操作按钮：
- 重新处理变量
- 重新读取初始变量
- 清除旧楼层变量
- 快照楼层 / 重演楼层

**设计启发**：这相当于给对话系统加了一个"存档系统"，每次对话的状态变化都是可追溯、可回退的。

### 2.2 ZOD 结构脚本 —— 数据模型（核心）

用 Zod schema 定义了整个游戏世界的数据结构，这是整张卡最精巧的部分：

```
Schema
├── 世界定位
│   ├── 当前大域/区域/场景
│   ├── 场景类型枚举（野外/宗门/城镇/秘境/战场/海域/地府/王朝/商会/酒家/禁地）
│   ├── 当前事件枚举（宗门大比/秘境/炼器/炼丹/奇遇/拍卖会/阵法/符箓/驭兽...）
│   ├── 当前状态枚举（正常/受伤/重伤/濒死/修炼/闭关/战斗/突破中/中毒/昏迷...）
│   └── 日期/天气
├── 在场人物 —— Record<角色名, {境界, 势力, 年龄}>
├── 用户信息
│   ├── 基础：姓名/性别/年龄/灵根/体质/称号/修真百艺
│   ├── 修为：境界枚举（凡人→炼气→筑基→...→真仙境十重天大圆满）
│   ├── 剑心境界枚举（入剑→剑意→剑势→剑心→...→无我无剑）
│   ├── 丹道境界枚举（炼精化气→炼气化神→炼神还虚→炼虚合道）
│   ├── 背包系统：灵石(四品)、丹药、材料、杂物
│   ├── 功法系统：主修/辅修/术法
│   ├── 法宝系统：本命/防御/辅助/未装备
│   ├── 灵兽 —— Record<灵兽名, {品种, 境界, 描述}>
│   └── 异火 —— Record<异火名, {品级, 属性, 描述}>
├── 男性好感度 —— Record<角色名, {好感度[-1000,1000], 关系阶段, 年龄, 重要事件}>
├── 女性好感度 —— 同上，分轨管理
└── 剧情标记 —— Record<string, string>
```

**设计启发**：
- 用 **枚举** 约束所有状态值，防止AI输出不合法的状态
- 用 `z.coerce.number().transform(v => _.clamp(v, 0, 100))` 做**值域约束**，确保数值不越界
- 用 `Record<string, object>` 实现**动态扩展**（灵兽、法宝等不限数量）
- 用 `.prefault({})` 提供默认值，确保结构完整
- 好感度分**男性/女性双轨**，暗示不同的关系发展逻辑

### 2.3 音乐播放器

完整的前端音乐播放器实现，功能包括：
- 从 AI 回复中解析 `<music>` 标签自动播放
- Catbox 文件托管
- 可视化播放器 UI（封面、进度条、音量控制）
- 拖拽、缩小为悬浮窗
- AudioContext 频谱分析实现动态光晕效果

**设计启发**：AI 的回复不仅是文本，还能触发多媒体体验。`<music>` 标签是一种简单而强大的扩展协议。

### 2.4 Evolution World Assistant（EW）

这是最核心的运行时引擎。它管理了两个**独立的 AI 工作流**（Flow）：

| Flow | 名称 | 职责 |
|------|------|------|
| Flow 1 | 角色更新 | 更新角色档案（人设、调色盘、三面性） |
| Flow 2 | 角色记忆 | 以角色主观视角更新记忆 |

每个 Flow 有独立的：
- Prompt 模板（完整的提示词编排）
- API 配置（支持独立的模型/参数）
- 生成参数（temperature: 1.2, frequency_penalty: 0.85）
- 上下文窗口（max_context_tokens: 2000000）

**关键参数选择**：
- `temperature: 1.2` —— 偏高，鼓励创造性输出
- `frequency_penalty: 0.85` —— 强力去重，避免重复描写
- `presence_penalty: 0.5` —— 中等，鼓励新话题但不失控
- `context_turns: 8` —— 只看最近8轮，控制成本

## 三、提示词模板层拆解

### 3.1 提示词编排顺序（Prompt Order）

以 Flow 1（角色更新）为例，完整编排如下：

```
1.  🐍 蛇娘防标记（depth:0） —— 随机token注入，防API限流
2.  🔓 Catbox破限（depth:4） —— 角色扮演框架 + 自检模块
3.  ⚙️  变量初始化（depth:4） —— 初始化模板变量
4.  🐍 秋青子身份（role:assistant, depth:4） —— 助手人格设定
5.  📐 创作原则-绝对零度（depth:4） —— 输出质量红线
6.  📝 输出格式要求EW适配（depth:4） —— JSON输出规范
7.  [World Info before] —— 世界书条目（触发词匹配）
8.  [Char Description] —— 角色描述
9.  [Persona Description] —— 玩家描述
10. [Char Personality] —— 角色性格
11. [Scenario] —— 情景
12. [World Info after] —— 世界书条目（后置）
13. [Chat Examples] —— 对话示例
14. 聊天记录开始 —— <chathistory> 标签
15. [Chat History] —— 实际聊天记录
16. 聊天记录结束 —— </chathistory>
17. 📋 动态世界记录条目加载说明
18. 📋 角色动态更新 —— 角色档案结构规范
19. 📋 角色记忆系统 —— 记忆写作规范
20. 📋 主时间线系统 —— 客观事件记录规范
21. 💭 秋青子开始思考 —— 强制 <thinking> 标签
22. 💭 角色更新思维链 —— 分步推理指令
23. 💭 记忆更新思维链
24. 💭 时间线更新思维链
25. 💭 controller_model配置思维链
26. 💭 秋青子思考结束 —— </thinking>
27.  输出格式（EW适配） —— JSON格式强制
28. 🔓 继续-Kemini —— 最后一条用户消息注入
29. 🌕 1 —— 人格锚定
30. 🌕 2 —— 复现指令
31. 🐍 秋青子吃掉了原生思维链（role:assistant） —— 预填充，劫持原生思维链
```

**设计要点**：
- **双重思维链劫持**：先用 `</think>` 关闭模型原生思维链，再用自定义 `<thinking>` 标签接管
- **depth 分层**：防标记在 depth:0（最外层），其余在 depth:4
- **role 切换**：大部分是 system，但秋青子身份和思维链劫持是 assistant role，实现预填充

### 3.2 防标记系统

```
{{random::🐍,🌙,💚,...}}{{random::秋,青,子,...}}{{random::a,b,...}}{{random::1,...}}
```

**原理**：每次请求注入随机 emoji+汉字+字母+数字组合，让每次请求的 token 序列都不同，防止 API 提供商通过 token 模式识别和限流（429）。

### 3.3 创作原则 —— "绝对零度"

这是一套**反AI八股**的写作规范：

| 禁止 | 替代方案 |
|------|---------|
| 模糊词（似乎、仿佛） | 具体事实 |
| 劣质比喻（像小兽） | 直接白描 |
| 微表情（嘴角上扬） | 行为描写 |
| 语气描写（带着xx的口吻） | 纯对话 |
| 过度心理描写 | 用行为展现 |
| 形容词堆砌 | 名词+动词 |

**核心思想**：Show, don't tell。用行为和对话展现性格，而非用修饰词描述。

### 3.4 EW 输出格式

AI 的输出不是自由文本，而是**结构化 JSON**：

```json
{
  "operations": {
    "worldbook": {
      "desired_entries": [
        {"name": "角色名", "content": "完整档案文本"}
      ],
      "remove_entries": []
    },
    "controller_model": {
      "template_id": "entry_selector_v1",
      "char_detection": {
        "alias_map": {"殷冬雪": "殷冬雪", "冬雪": "殷冬雪"},
        "scene_var": "stat_data.在场人物",
        "scan_messages": 1,
        "entry_patterns": ["{name}", "{name}_记忆"]
      },
      "fallback_entries": ["主时间线"]
    }
  }
}
```

EW 插件解析这个 JSON 后，自动将内容写入世界书条目，并通过 controller_model 控制哪些条目加载到主对话的上下文中。

## 四、角色人设方法论拆解

这张卡定义了一套完整的**角色创建方法论**，是最有学习价值的部分。

### 4.1 角色档案四部曲

```
一、基础信息 —— 绝对零度白描（200-400字）
二、性格调色盘 —— 衍生展开（800-2000字）
三、三面性 —— 按需使用（每面300-500字）
四、对角色的理解与思考 —— 二次解释（300-800字）
```

### 4.2 性格调色盘（核心创新）

**概念**：人的性格像调色盘，由底色+主色调+点缀组成，通过"衍生"展开。

```
底色 —— 最底层的核心驱动（如：呆萌）
主色调 —— 日常最常表现的性格（如：认真、认死理）
点缀 —— 特定情境下浮现的性格（如：格格不入的纯粹）
```

**衍生规则**：
- 不写"她很温柔"，写她在具体场景做了什么
- 衍生之间可以矛盾（这才是真实的人）
- 每个性格至少2-3个衍生，最多5个
- 可以对冲（"社恐"+"社交达人面具"）

**实例**（沈艺清）：
```
呆萌衍生一：周围师姐在讨论如何迷倒男人，她蹲在角落炼毒。
呆萌衍生二：别人说"今晚月色真美"，她说"确实很亮，适合炼毒"。
认真衍生一：毒鼎里每一种毒了如指掌，每个配方反复验证几百遍。
认死理衍生二：为验证上古毒方，把自己中毒了37次。
```

### 4.3 三面性

**判断标准**：角色在不同场景下，行为模式是否有**根本性切换**。

每张面的五个部件：
1. 触发条件
2. 能量状态
3. 语料（5-10句具体台词）
4. 身体行为模式
5. 功能（这张面在保护什么？）

面与面之间必须写**过渡**和**渗透**。

### 4.4 二次解释（防AI脑补）

**功能**：告诉AI"这个性格是这个意思，不是你以为的意思"。

```
关于乐观的双重性：
她的乐观在于和{{user}}在一起时是真实自然的，而非全都是虚假的。
但和非{{user}}的人相处时，乐观就是实打实的假象。
这不意味着她没有伤心的时候，只是脆弱时她会直接表现在{{user}}面前。
```

## 五、记忆系统拆解

### 5.1 核心理念

> 记忆不是客观记录，是角色的主观体验。

同一顿饭：
- 喜欢你的角色："他给我倒了水，手指碰到杯子的时候犹豫了一下。"
- 讨厌你的角色："又在装好人。倒水的时候特意看了我一眼，恶心。"
- 无所谓的角色：根本不记这顿饭。

### 5.2 记忆形式（5种）

| 形式 | 适合角色 | 示例 |
|------|---------|------|
| 物品关联型 | 有收集癖/情感寄托 | "银色发卡：是他在夏祭上买的..." |
| 日记型 | 内向/内心戏丰富 | "三月初的某天：今天和他去了旧书店..." |
| 片段闪回型 | 创伤经历/直觉型 | "咖啡的味道：那天的咖啡是苦的..." |
| 关系定位型 | 社交敏感/警惕性强 | "关于角色B：一开始觉得她很碍眼..." |
| 碎念型 | 大大咧咧/想法直白 | "上周还是上上周来着，打了一场篮球..." |

### 5.3 记忆衰退与篡改

**三级衰退**：
```
近期 → 细节清晰，情感鲜活
中期 → 细节模糊，情感沉淀为判断
远期 → 只留核心，可能失真
```

**记忆篡改**：
- 关系改善 → 负面记忆被重新解读
- 关系恶化 → 正面记忆被扭曲
- 创伤 → 记忆封锁/美化/替换

**容量控制**：
- 近期：最多8条（50-150字/条）
- 沉淀：最多8条（20-60字/条）
- 核心：最多5条（20-80字/条）
- 单角色总计：500-1500字

### 5.4 非全知原则

角色只能记住自己看到的、听到的、感受到的。不在场的事不记。

## 六、时间线系统拆解

### 6.1 与记忆系统的分工

```
时间线 —— "发生了什么"（客观事实）
记忆 —— "怎么理解"（主观体验）
人设 —— "因此变成了什么人"（性格演化）
```

### 6.2 事件三级分类

| 级别 | 类型 | 处理方式 |
|------|------|---------|
| A级 | 转折点（告白、背叛、质变） | 永不压缩 |
| B级 | 推进点（新信息、新联系） | 远期压缩为一句话 |
| C级 | 填充（日常闲聊） | 不记录 |

### 6.3 时间线结构

```
当前态势 —— 此刻的状态快照（50-100字）
活跃线索 —— 未解决的悬念（最多8条）
已发生事件 —— 按时间倒序的事件记录
```

## 七、世界书层拆解

共 **443条条目**，按关键词触发，覆盖：

- **角色人设**：每个角色遵循四部曲结构（基础信息+调色盘+三面性+二次解释）
- **势力设定**：宗门、家族、组织的层级和关系
- **地理世界观**：大域、区域、场景
- **功法体系**：修为境界、剑道、丹道
- **角色速览**：每个势力一张速查卡

条目特点：
- 使用 `selective: true` + 正则匹配关键词
- `depth: 4` 注入到聊天记录第4层
- `exclude_recursion: true` 防止递归触发
- `position: before_char` 放在角色描述之前

## 八、对我们项目的启发

### 可直接借鉴的设计：

1. **ZOD Schema 思路** → 在 Unity 中用 C# 类定义世界状态结构，用枚举约束所有状态值
2. **性格调色盘方法论** → 用于 PersonaManager 的 NPC 人设编写规范
3. **记忆系统** → 直接指导 NPCMemoryStorage 的设计：主观视角、衰退机制、容量控制
4. **时间线系统** → 作为 WorldBookManager 的一个全局条目，维护客观事件线
5. **EW 的 controller_model** → 我们的 Context Assembly 系统可以参考其条目加载逻辑
6. **创作原则"绝对零度"** → 作为系统提示词的写作质量约束
7. **双重思维链** → 在 Author's Note 中引导 AI 先推理再输出

### 架构差异：

| 酒馆方案 | 我们的 Unity 方案 |
|---------|-----------------|
| 前端脚本 + 世界书 | C# MonoBehaviour + ScriptableObject |
| JSON 输出 → 解析写入 | 直接 C# 对象操作 |
| 多 Flow 并行调用 | 单次 API 调用 + 本地后处理 |
| 443条世界书条目 | WorldBookManager 按关键词检索注入 |
| 文本变量持久化 | PlayerPrefs / 存档文件 |
